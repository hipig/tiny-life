# AGENTS.md

本项目是一个 Godot 4.x / GDScript 手游项目，当前项目起点以 `docs/PRD.md` 和 `docs/TRD.md` 为准。

项目暂定名：

- 中文：`小小公寓物语`
- 英文：`Tiny Apartment Story`

核心概念：

- 竖屏像素风休闲公寓经营游戏。
- 公寓楼本身就是主界面。
- 玩家扩建楼层、装修房间、招募租客、自动获得租金，并观察小人在自己布置的房间里生活。

## 开始前必须阅读

在进行产品、设计、玩法、架构或内容修改前，先阅读：

1. `docs/PRD.md`
2. `docs/TRD.md`
3. `.agents/rules/` 下的规则文件

PRD/TRD 是项目源头约束。如果实现压力与文档意图冲突，优先保护产品体验，并在变更说明中标明取舍。

## 规则文件

- `.agents/rules/product.md`：产品定位、MVP 范围、禁止方向。
- `.agents/rules/technical.md`：Godot/GDScript 架构、场景脚本布局、平台抽象、存档、性能规则。
- `.agents/rules/interaction.md`：UI 状态机、输入优先级、房间/家具/建造交互规则。
- `.agents/rules/data-config.md`：配置表驱动、JSON 数据要求、经济与内容约束。

## 开发原则

- 使用 Godot 4.x 和 GDScript，不引入 C#。
- 优先按竖屏手机设计，逻辑视口基准为 `360 x 640`，桌面预览为 `720 x 1280` 整数放大。
- 主界面必须围绕公寓场景，不添加传统底部常驻导航。
- 优先通过场景交互完成操作，而不是堆叠菜单入口。
- 系统必须配置表驱动，不在业务代码中硬编码价格、奖励、租客倍率、家具属性、楼层成本或 UI 文案。
- 平台 SDK 必须隔离在 `PlatformManager`、`AdManager` 和 Provider 类后面。
- 存档读写必须通过 `SaveManager`，玩法代码不能直接依赖本地文件路径。
- 跨系统更新优先通过 `GameEvents` 信号/事件，不做紧耦合调用。
- 项目当前处于开发阶段，除非用户特别要求，不需要兼容旧配置或旧存档；涉及数据结构或配置含义调整时，直接更新配置、默认状态和测试。
- MVP 内容可以少，但框架要按可发行手游的形态搭建。

## MVP 里程碑顺序

优先按以下顺序推进：

1. 主公寓视图：竖屏布局、顶部状态栏、右侧悬浮按钮、相机拖动缩放、房间点击。
2. 房间 UI 与新家具摆放：房间面板、家具 Tab、家具商店、网格吸附、确认后购买。
3. 已有家具操作：长按、移动、取消移动、回收返还。
4. 租客招募与基础小人 AI。
5. 经济与等级成长：自动租金、金币提示合并、房间租金、公寓等级、建造槽。
6. 任务、离线收益、存档读写、激励广告抽象、平台 Mock。

## 质量要求

- 第一屏必须像一个温馨的像素公寓，而不是菜单大厅。
- 每个核心玩法流程都应回到公寓场景。
- UI 必须考虑手机安全区和竖屏长宽比。
- 避免每个家具或租客都运行不必要的逐帧逻辑。
- 重要状态变化后必须存档：家具摆放/移动/回收、租客入住、楼层建造、任务完成、应用暂停/退出。
- 如果新增系统会影响租金、房间评分、租客满意度、任务进度或存档数据，必须同步更新相关 Manager 与事件。
